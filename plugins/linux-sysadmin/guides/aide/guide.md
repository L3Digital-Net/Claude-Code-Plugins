# aide

> **Based on:** aide 0.19.3 | **Updated:** 2026-03-27

## Identity
- **Binary**: `/usr/bin/aide`
- **Config**: `/etc/aide/aide.conf` (Debian/Ubuntu), `/etc/aide.conf` (RHEL/Fedora)
- **Database (input)**: `/var/lib/aide/aide.db` (what AIDE checks against)
- **Database (output)**: `/var/lib/aide/aide.db.new` (generated by `--init` or `--update`)
- **Logs**: stdout by default; redirect with `report_url = file:/var/log/aide/aide.log`
- **Cron wrapper**: `/etc/cron.daily/aide` (Debian ŌĆö runs nightly check automatically)
- **Install**: `apt install aide` / `dnf install aide`

## Quick Start

```bash
sudo apt install aide
sudo aideinit                         # Debian wrapper: inits DB and copies to aide.db
# ŌĆö or on RHEL/manual ŌĆö
sudo aide --init                      # creates aide.db.new
sudo cp /var/lib/aide/aide.db.new /var/lib/aide/aide.db
sudo aide --check                     # compare current filesystem against database
```

## Key Operations

| Task | Command |
|------|---------|
| Initialize database | `sudo aide --init` (then copy `.db.new` to `.db`) |
| Check integrity | `sudo aide --check` |
| Update database (check + new baseline) | `sudo aide --update` |
| Compare two databases | `sudo aide --compare` |
| Verbose/debug output | `sudo aide --verbose=255` |
| Dry-run config test | `sudo aide --config-check` |
| Debian auto-init | `sudo aideinit` |

## How It Works

```
1. aide --init
   Scans filesystem per aide.conf rules ŌåÆ writes aide.db.new

2. Copy aide.db.new ŌåÆ aide.db
   This becomes the trusted baseline

3. aide --check
   Scans filesystem again ŌåÆ compares against aide.db ŌåÆ reports differences

4. aide --update
   Same as --check but also writes a new aide.db.new for the next cycle
```

The cycle is: init once, check regularly, update the baseline after reviewing changes.

## Configuration Concepts

Selection lines control which paths AIDE monitors:

```
# Regular selection ŌĆö monitor /bin with rule R
/bin    R

# Equals selection ŌĆö only the exact file, not children
=/var/log/messages$  R+a

# Negative selection ŌĆö exclude from monitoring
!/var/log/.*
!/tmp
!/proc
!/run
```

Predefined rule groups:

| Rule | Meaning |
|------|---------|
| `R` | `p+ftype+i+l+n+u+g+s+m+c+md5` ŌĆö read-only files |
| `L` | `p+ftype+i+l+n+u+g` ŌĆö log files (skip content hashes) |
| `>` | `p+ftype+l+u+g+i+n+S` ŌĆö growing files (size only increases) |
| `E` | Empty group ŌĆö track existence only |

Custom rules combine attributes: `MyRule = p+i+n+u+g+s+b+m+c+sha256+sha512`

## Common Failures

| Symptom | Cause | Fix |
|---------|-------|-----|
| Every check reports thousands of changes | Monitoring volatile paths (`/tmp`, `/var/cache`, `/proc`) | Add exclusions: `!/tmp` `!/proc` `!/var/cache` |
| `aide --check` says "database not found" | Never initialized or wrong `database_in` path | Run `aide --init` then copy `.db.new` to `.db`; check `database_in` in config |
| Config syntax error on startup | Malformed selection line or unknown attribute | Run `aide --config-check` to pinpoint the error |
| Update clobbers your baseline | Forgot to copy `.db.new` ŌåÆ `.db` after update | Always review changes before promoting the new database |
| Debian cron report is empty | `aideinit` was never run after install | Run `sudo aideinit` to create the initial database |
| False positives on package upgrades | Expected: system binaries changed | Run `aide --update` after verified upgrades and promote the new database |

## Pain Points

- **Database promotion is manual by design.** AIDE never overwrites `aide.db` automatically. After `--init` or `--update`, you must copy `aide.db.new` to `aide.db` yourself. This is intentional so you review changes before trusting a new baseline.

- **Debian's `aideinit` vs raw `aide --init`**: On Debian/Ubuntu, `aideinit` is a wrapper that runs `--init` and copies the database for you. On RHEL or manual installs, you do the copy yourself.

- **Exclude volatile paths early.** Monitoring `/var/log`, `/tmp`, `/proc`, or `/run` generates massive noise. Use the `L` (log) or `>` (growing) rule for log files, and `!` exclusions for truly volatile trees.

- **Hash algorithm choice matters for performance.** `md5` is fast but weak; `sha256+sha512` is strong but doubles scan time on large filesystems. Pick one strong hash (`sha256`) unless compliance requires multiple.

- **Store the database off-host for real security.** An attacker with root can modify both the filesystem and the AIDE database. For genuine integrity assurance, keep `aide.db` on read-only media or a separate host.

- **Cron reports need a mail relay.** The Debian daily cron job mails its report to root. If no MTA is configured (or root mail isn't forwarded), reports go nowhere. Either configure `msmtp`/`postfix` or set `report_url` to a file and monitor it.
