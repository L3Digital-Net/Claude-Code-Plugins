# Mail Stack DNS Records

Complete DNS record set for a self-hosted mail server. All examples use
`example.com` as the domain and `203.0.113.10` as the server IP.
Replace with your actual values.

---

## Record Summary

| Record | Host/Name | Type | Value |
|--------|-----------|------|-------|
| MX | `example.com` | MX | `10 mail.example.com.` |
| A | `mail.example.com` | A | `203.0.113.10` |
| PTR | `10.113.0.203.in-addr.arpa` | PTR | `mail.example.com.` |
| SPF | `example.com` | TXT | `v=spf1 mx -all` |
| DKIM | `mail._domainkey.example.com` | TXT | `v=DKIM1; k=rsa; p=<public-key>` |
| DMARC | `_dmarc.example.com` | TXT | `v=DMARC1; p=reject; rua=...` |
| Autoconfig | `autoconfig.example.com` | CNAME | `mail.example.com.` |

---

## MX Record

Tells sending servers where to deliver mail for `@example.com` addresses.

```
example.com.    IN    MX    10    mail.example.com.
```

- **Priority 10** is standard for a single mail server. Lower number = higher priority.
- The target (`mail.example.com.`) must have an A record; MX targets cannot be CNAMEs or bare IPs.
- For backup MX, add a second record with a higher priority number (e.g., `20 backup-mx.example.com.`).

Verify: `dig +short MX example.com` should return `10 mail.example.com.`

## A Record

Resolves the mail server hostname to its IP address.

```
mail.example.com.    IN    A    203.0.113.10
```

If you also use IPv6, add an AAAA record:

```
mail.example.com.    IN    AAAA    2001:db8::1
```

Verify: `dig +short A mail.example.com` should return `203.0.113.10`

## PTR Record (Reverse DNS)

Maps the server IP back to the hostname. Receiving servers check that the connecting
IP resolves to a name matching the SMTP EHLO hostname. Mismatch causes spam scoring
or outright rejection.

```
10.113.0.203.in-addr.arpa.    IN    PTR    mail.example.com.
```

**This record is set at your hosting/VPS provider, not your DNS registrar.** Each
provider has a different interface for it:
- Hetzner: Cloud Console → Server → Networking → Reverse DNS
- DigitalOcean: Droplet name must match the desired PTR (set via API or rename)
- AWS: Submit a form to AWS support for EIP reverse DNS
- Linode: Networking tab → Reverse DNS

Verify: `dig -x 203.0.113.10 +short` should return `mail.example.com.`

## SPF Record

Specifies which servers are authorized to send mail for your domain. Receiving
servers check the envelope sender's IP against this record.

```
example.com.    IN    TXT    "v=spf1 mx -all"
```

Breakdown:
- `v=spf1` — version identifier (required)
- `mx` — authorize the IPs that the domain's MX records resolve to
- `-all` — reject (hard fail) mail from any other source

Variations:
- `v=spf1 ip4:203.0.113.10 -all` — authorize a specific IP
- `v=spf1 mx a:mail.example.com -all` — authorize MX IPs and a specific hostname
- `v=spf1 mx include:_spf.google.com -all` — also authorize Google Workspace
- `~all` (soft fail) instead of `-all` (hard fail) is more lenient but weaker

Only one SPF record per domain. Multiple `v=spf1` records cause SPF to fail entirely.

Verify: `dig +short TXT example.com | grep spf`

## DKIM Record

Publishes the public key that receiving servers use to verify the DKIM signature
added by OpenDKIM. The record name includes the selector (here: `mail`).

```
mail._domainkey.example.com.    IN    TXT    "v=DKIM1; k=rsa; p=MIIBIjANBgkq..."
```

- The public key is generated by `opendkim-genkey`. The file `mail.txt` in the key
  directory contains the exact DNS record value.
- The selector (`mail`) matches the `-s` flag used during key generation and the
  `Selector` directive in `opendkim.conf`.
- Some DNS providers split long TXT records into multiple 255-byte strings. This is
  fine as long as the provider joins them correctly. Test with `dig` to verify the
  full value is returned.

To get the record value from the server:
```bash
sudo cat /etc/opendkim/keys/example.com/mail.txt
```

Verify:
```bash
dig +short TXT mail._domainkey.example.com
opendkim-testkey -d example.com -s mail -vvv
```

### DKIM Key Rotation

To rotate keys without downtime:
1. Generate a new key pair with a new selector (e.g., `mail2`).
2. Publish the new selector's DNS TXT record.
3. Wait for DNS propagation.
4. Update OpenDKIM config to sign with the new selector.
5. Remove the old selector's DNS record after its TTL expires.

## DMARC Record

Ties SPF and DKIM together with a policy and reporting address. Receiving servers
apply this policy when either SPF or DKIM fails or does not align with the From header.

```
_dmarc.example.com.    IN    TXT    "v=DMARC1; p=reject; rua=mailto:dmarc-reports@example.com; ruf=mailto:dmarc-forensics@example.com; adkim=r; aspf=r; pct=100"
```

Breakdown:
- `v=DMARC1` — version (required)
- `p=reject` — policy for failures: `none` (monitor only), `quarantine` (spam folder), `reject` (bounce)
- `rua=mailto:...` — aggregate reports sent here (daily XML digests)
- `ruf=mailto:...` — forensic reports for individual failures (optional; not all providers send these)
- `adkim=r` — DKIM alignment mode: `r` (relaxed, subdomains OK) or `s` (strict, exact match)
- `aspf=r` — SPF alignment mode: same options
- `pct=100` — percentage of failing messages to apply the policy to

**Recommended rollout:**
1. Start with `p=none` to collect reports without blocking mail.
2. Review aggregate reports for a few weeks to catch legitimate senders not covered by SPF/DKIM.
3. Move to `p=quarantine` for a week.
4. Move to `p=reject` once confident.

Verify: `dig +short TXT _dmarc.example.com`

## Autoconfig / Autodiscover (Optional)

These records help mail clients (Thunderbird, Outlook) auto-detect server settings.

For Thunderbird (autoconfig):
```
autoconfig.example.com.    IN    CNAME    mail.example.com.
```

Then serve an XML file at `https://autoconfig.example.com/mail/config-v1.1.xml`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<clientConfig version="1.1">
  <emailProvider id="example.com">
    <domain>example.com</domain>
    <incomingServer type="imap">
      <hostname>mail.example.com</hostname>
      <port>993</port>
      <socketType>SSL</socketType>
      <authentication>password-cleartext</authentication>
      <username>%EMAILADDRESS%</username>
    </incomingServer>
    <outgoingServer type="smtp">
      <hostname>mail.example.com</hostname>
      <port>587</port>
      <socketType>STARTTLS</socketType>
      <authentication>password-cleartext</authentication>
      <username>%EMAILADDRESS%</username>
    </outgoingServer>
  </emailProvider>
</clientConfig>
```

For Outlook (autodiscover), publish an SRV record:
```
_autodiscover._tcp.example.com.    IN    SRV    0 0 443 mail.example.com.
```

## Verification Checklist

Run these after all records are in place:

```bash
# MX
dig +short MX example.com
# Expected: 10 mail.example.com.

# A
dig +short A mail.example.com
# Expected: 203.0.113.10

# PTR
dig -x 203.0.113.10 +short
# Expected: mail.example.com.

# SPF
dig +short TXT example.com | grep spf
# Expected: "v=spf1 mx -all"

# DKIM
dig +short TXT mail._domainkey.example.com
# Expected: "v=DKIM1; k=rsa; p=..."

# DMARC
dig +short TXT _dmarc.example.com
# Expected: "v=DMARC1; p=reject; ..."
```

Or use MXToolbox for a comprehensive check: https://mxtoolbox.com/SuperTool.aspx
